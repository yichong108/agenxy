# 目标

让一次发布具备可追踪、可验证、可回滚能力。

# 适用场景

- 用户要求“发版、上线、打包、发布前检查”。
- 需要汇总改动、执行质量门禁并给出发布指令。

# 执行流程

1. 变更确认
   - 列出本次改动范围与影响模块。
   - 明确是否包含破坏性变更。

2. 质量门禁
   - 依次执行：测试、类型检查、构建（按项目实际命令）。
   - 任一步骤失败则停止发布，先处理失败项。

3. 发布动作
   - 给出标准发布命令与执行顺序。
   - 标记关键产物（版本号、构建产物、变更摘要）。

4. 回滚预案
   - 明确回滚触发条件和回滚步骤。
   - 说明回滚后验证点。

# 工具映射建议

- 查看改动与上下文：`search_workspace`、`read_file`
- 记录发布说明：`write_file`
- 运行门禁与发布命令：`shell`

# 输出结构

1. 发布范围与风险等级
2. 门禁执行结果（测试/类型/构建）
3. 发布步骤与命令
4. 回滚方案
5. 发布后观察项
