# Debug Workflow 参考规范

## 根因判断标准

- 根因应满足“修复后同类症状消失”，而不只是当前输入下不再报错。
- 若只能缓解症状，需要在结论中明确“临时修复”并给出后续方案。
- 存在多个可疑点时，优先选择证据最强、改动最小、可验证性最高的路径。

## 变更范围控制

- 仅允许修改与根因直接相关的文件和逻辑分支。
- 避免“顺手重构”；格式化、重命名、结构调整应与修复分开提交。
- 若必须扩大改动面，先在输出中说明原因与影响范围。

## 风险分级

- `低`：仅局部逻辑修复，已有测试覆盖，验证通过。
- `中`：涉及公共函数或关键路径，验证项有限。
- `高`：涉及接口行为变化、数据结构变化或无法完整验证。

## 验证基线

- 至少执行一项“最贴近问题”的验证（单测/命令/手动复现）。
- 若项目支持，补充类型检查或构建验证。
- 无法本地验证时，必须列出未验证项和建议验证命令。

## 输出质量要求

- 结论可追溯：每个判断都应能对应到复现现象或代码证据。
- 描述可执行：给出可直接复制的命令或步骤。
- 结论可交接：他人读完即可继续处理，无需二次猜测。
