# 目标

在不引入回归的前提下，完成可验证的最小修复。

# 执行策略

1. 先理解问题
   - 明确报错现象、触发条件、预期行为。
   - 若信息不足，先读取相关文件并给出最小可行假设。

2. 再做定位
   - 优先读取入口文件和调用链关键节点。
   - 对可疑分支进行对比，找出根因而不是只修表象。

3. 最小化改动
   - 仅修改与根因直接相关代码，避免无关重构。
   - 维持现有接口与行为兼容，除非用户明确要求变更。

4. 修后验证
   - 运行最贴近问题的验证命令（测试、类型检查或构建）。
   - 给出验证结果与剩余风险。

# 工具使用建议

- 先 `read_file` / `list_dir` / `search_workspace`，后 `write_file`。
- 需要执行验证时使用 `shell`。
- 避免一次性大面积改写，优先小步快跑。

# 输出要求

1. 根因说明（简洁）
2. 修改点（文件与改动意图）
3. 验证步骤与结果
4. 若仍有风险，明确列出后续建议

# 安全与质量约束

- 不泄露密钥、令牌、凭据等敏感信息。
- 不访问工作区外路径。
- 不执行明显危险或破坏性命令。
